# 发布与打包

这份文档聚焦 **发布者 / 维护者视角** 的打包说明。

如果你只是想从源码运行，或者先生成一个本地体验包，请优先阅读：

- [从源码构建与打包](从源码构建与打包.md)

## 打包模式

Caps Nav 当前提供两种打包模式：

- `dev`
  - 用于本地测试分发
  - 基于 `Release` 配置构建
  - 不包含 `Developer ID` 正式签名、公证和票据贴附
- `release`
  - 用于正式发布，包含签名、公证和最终验证

这里要特别注意：

- `dev` 不是 `Debug` 包
- `dev` 也不是 Xcode 意义上的“开发包”
- 它更准确地说，是一个“未完成正式签名与公证的 Release 级本地分发包”

## 目标产物

当前脚本会生成：

- `Universal` 通用版 App
  - 同时包含 `arm64` 与 `x86_64`
- 一个可分发的安装包（`.dmg`）
- 一个对应的 `SHA-256` 摘要文件

最低系统版本仍受工程配置约束，目前为：

- `macOS 13.0+`

## 前提条件

执行打包前，建议先确认：

- 本机已安装 Xcode 和 Command Line Tools（命令行开发工具）
- 当前工程能正常构建
- 可以执行 `xcodebuild`
- 可以执行 `swift`
- 系统可用 `hdiutil`、`ditto`、`shasum`、`osascript`

如果你要执行 `release` 模式，还需要：

- 钥匙串里已安装可用的 `Developer ID Application` 证书（Apple 开发者分发签名证书）
- 已配置 `xcrun notarytool` 的 keychain profile（公证凭据配置）
- 本机可用 `codesign`、`xcrun`、`spctl`

## 一条命令打包

### `dev` 模式

```bash
./tools/package.sh --mode dev
```

这个模式适合：

- 你自己本机验证安装流程
- 发给少量测试用户体验功能
- 还没有配置 Apple 开发者正式分发能力时先把安装包产出来

脚本会自动完成：

1. 构建 `Release`
2. 生成 `arm64 + x86_64` 通用版 App
3. 校验双架构
4. 生成视觉化安装包（DMG）
5. 生成 `.dmg.sha256`

默认输出目录：

```text
artifacts/dev/
```

### `release` 模式

```bash
./tools/package.sh --mode release \
  --signing-identity "Developer ID Application: Example Corp (TEAM123456)" \
  --team-id "TEAM123456" \
  --notary-profile "capsnav-release"
```

脚本会自动完成：

1. `xcodebuild archive`
2. 生成 `ExportOptions.plist`
3. `xcodebuild -exportArchive`
4. 校验签名后的导出 App
5. 生成最终安装包（DMG）
6. `xcrun notarytool submit --wait`
7. `xcrun stapler staple`
8. `xcrun stapler validate`
9. `spctl` 最终验证
10. 生成 `.dmg.sha256`

## 常用参数

### 自定义输出目录

```bash
./tools/package.sh --mode dev --output-dir /absolute/path/to/output
```

### 自定义 DerivedData 目录

```bash
./tools/package.sh --mode dev --derived-data /absolute/path/to/.build/ReleasePackaging
```

### 跳过 SHA-256 摘要

```bash
./tools/package.sh --mode dev --skip-sha256
```

## `release` 模式的参数来源

`release` 模式必须拿到这三个值：

- `signing identity`
- `team id`
- `notary profile`（公证凭据名称）

脚本读取优先级是：

1. 命令行参数
2. 环境变量
3. 缺失时报错退出

对应环境变量：

```bash
export CAPS_NAV_SIGNING_IDENTITY="Developer ID Application: Example Corp (TEAM123456)"
export CAPS_NAV_TEAM_ID="TEAM123456"
export CAPS_NAV_NOTARY_PROFILE="capsnav-release"
```

然后可以直接执行：

```bash
./tools/package.sh --mode release
```

## `notarytool` 凭据建议

建议先在发布机器上保存一次凭据：

```bash
xcrun notarytool store-credentials "capsnav-release" \
  --apple-id "your-account@example.com" \
  --team-id "TEAM123456" \
  --password "app-specific-password"
```

之后脚本里只传 `--notary-profile` 即可，不要把真实 Apple ID 和密码写进仓库。

## GitHub Actions 手动生成 dev 发布产物与更新元数据

如果你想在 GitHub 上手动生成一个可下载的 `dev` 安装包，并同时更新客户端检查更新所需的元数据，仓库已经提供：

- [`.github/workflows/pre-release.yml`](../../.github/workflows/pre-release.yml)

如果你想在 GitHub 上自动触发一个“带 Developer ID 签名、公证、票据贴附”的正式包流程，仓库已新增：

- [`.github/workflows/signed-release.yml`](../../.github/workflows/signed-release.yml)

这条工作流的定位是：

- 以你手动创建的 GitHub `Release + tag` 为发布事实源
- 自动构建对应 tag 的 `dev` 安装包
- 自动上传 `DMG` 和 `SHA-256` 文件到这个已有 Release
- 在需要时更新 `docs/updates/latest.json`
- 让客户端通过 GitHub Pages 读取最新版本信息和 Markdown 更新说明

这里生成的仍然是 `dev` 包，也就是：

- 基于 `Release` 配置构建
- 不依赖本地 Apple ID 登录
- 不要求 Apple Developer Program 年费账号
- 不包含 `Developer ID` 正式签名、公证和票据贴附
- 当前 workflow 会尽量保留与本地 `dev` 打包一致的 DMG 视觉布局，并默认走 `--headless-dmg`

### 这条 workflow 的边界

这条工作流的目标是“稳定自动出包 + 下发更新元数据”，不是“生成最终可公开分发的正式安装盘”，所以它有这些限制：

- 第一次打开时，macOS 很可能弹出安全拦截提示
- 用户需要手动到“系统设置 -> 隐私与安全性 -> 安全性”中选择“仍要打开”
- 它仍然不是经过正式签名、公证和票据贴附的最终公开发布包
- 它适合提前体验、测试、灰度分发，不适合作为最终正式发布形态
- 它不会替你生成 Release Notes，客户端展示什么内容，取决于你在 GitHub Release 里手写了什么 Markdown

## GitHub Actions 自动生成 signed 正式包

`signed-release.yml` 的定位是：

- 默认由 `release.published` 自动触发
- 保留 `workflow_dispatch` 入口用于手动补跑
- 基于你已存在的 `Release tag` 构建正式包
- 走 `./tools/package.sh --mode release`
- 自动执行 `Developer ID` 签名、公证、票据贴附
- 将产物上传到对应 Release（同名资产会覆盖旧文件）
- 可选更新 `docs/updates/latest.json`

### 需要先配置的 GitHub Secrets

在仓库 `Settings -> Secrets and variables -> Actions` 中新增：

- `CAPS_NAV_DEVELOPER_ID_P12_BASE64`
  - `Developer ID Application` 证书导出的 `.p12` 文件做 base64 后的内容
- `CAPS_NAV_DEVELOPER_ID_P12_PASSWORD`
  - 导出 `.p12` 时设置的密码
- `CAPS_NAV_SIGNING_IDENTITY`
  - 例如：`Developer ID Application: Example Corp (TEAM123456)`
- `CAPS_NAV_TEAM_ID`
  - Apple Developer Team ID，例如：`TEAM123456`
- `CAPS_NAV_NOTARY_APPLE_ID`
  - 公证使用的 Apple ID（邮箱）
- `CAPS_NAV_NOTARY_APP_PASSWORD`
  - 该 Apple ID 对应的 app-specific password

### 触发方式

1. 自动触发（推荐）

- 创建并发布 GitHub Release（事件：`release.published`）
- 系统自动运行 `Publish Signed Release Assets`

2. 手动补跑（可选）

- 打开仓库 `Actions`
- 选择 `Publish Signed Release Assets`
- 点击 `Run workflow`
- 输入：
  - `branch`：用于校验 tag 所属历史（默认 `main`）
  - `tag`：目标 Release tag，例如 `v0.0.3`
  - `publish_latest`：是否覆盖 `docs/updates/latest.json`

### 与 `pre-release.yml` 的关系

- `pre-release.yml`
  - 仅手动触发（`workflow_dispatch`）
  - 生成 `dev` 包
  - 不包含正式签名/公证
- `signed-release.yml`
  - 默认自动触发（`release.published`）
  - 支持手动补跑
  - 生成正式签名/公证包
  - 适合对外正式分发

### 使用方式

1. 先在工程里更新版本号，确保 `MARKETING_VERSION` 使用严格三段数字，例如 `0.0.2`
2. 推送代码到 GitHub 默认分支
3. 手动创建同名 tag，例如 `v0.0.2`
4. 手动创建 GitHub Release，并写好面向用户的 Markdown 更新说明
5. 发布这个 Release
6. 等待 `Publish Signed Release Assets` workflow 完成
7. 到仓库的 `Releases` 页面下载产物
8. 如已启用 GitHub Pages 的 `main /docs` 发布方式，客户端会从 `docs/updates/latest.json` 自动读到这次版本

### 版本号与标签规则

- Git tag 必须和工程里的 `MARKETING_VERSION` 对应
- 约定格式为 `v<version>`，例如：
  - `v0.0.1`
  - `v0.0.2`
- 客户端更新比较只使用严格三段数字版本号，例如 `0.0.2`
- workflow 会从构建产物里读取 `CFBundleShortVersionString` 写入 `docs/updates/latest.json`

如果你手动补跑 `workflow_dispatch`，workflow 会比较候选版本与当前 `docs/updates/latest.json` 里的版本号；只有更高版本才允许覆盖。

### 自动触发与手动补跑

`signed-release.yml` 支持两种入口：

1. `release.published`

- 默认把 `main` 上新建的 Release 视为最新版本
- 自动上传正式签名产物
- 自动更新 `docs/updates/latest.json`

2. `workflow_dispatch`

- 手动输入 `branch`
- 手动输入 `tag`
- 可选 `publish_latest`
- 适合补传资产、补写 `latest.json` 或修复历史 Release

手动补跑默认不覆盖 `latest.json`。WHY 是防止你补跑旧版本时，把客户端更新源回滚到历史版本。

`pre-release.yml` 目前只保留 `workflow_dispatch`，用于需要 `dev` 包时手动执行。

### GitHub Pages 与 `latest.json`

这套在线更新依赖 `main/docs` 发布到 GitHub Pages。

推荐仓库设置：

- `Settings -> Pages`
- `Source = Deploy from a branch`
- `Branch = main`
- `Folder = /docs`

当前更新元数据目录在：

- [docs/updates/README.md](../updates/README.md)
- `docs/updates/latest.json`

客户端默认读取的地址是：

```text
https://slowslicing.github.io/CapsNav.macOS/updates/latest.json
```

`latest.json` 至少包含这些字段：

- `version`
- `publishedAt`
- `minimumSystemVersion`
- `pageURL`
- `downloadURL`
- `notesMarkdown`

其中 `notesMarkdown` 会直接使用你在 GitHub Release 里手写的 Markdown 说明。

## 产物命名

当前产物命名格式为：

```text
Caps-Nav-<版本号>-<构建号>-macOS-universal.dmg
```

例如：

```text
Caps-Nav-1.0-1-macOS-universal.dmg
```

对应摘要文件：

```text
Caps-Nav-1.0-1-macOS-universal.dmg.sha256
```

## 安装包（DMG）内容

生成的安装包默认包含：

- `Caps Nav.app`
- `Applications` 快捷方式

用户打开安装包后，可以直接把 App 拖到“应用程序”目录。

## 视觉化安装盘

当前安装包会额外完成这些处理：

- 自动生成品牌化背景图
- 设置访达（Finder）窗口尺寸
- 固定 `Caps Nav.app` 与 `Applications` 的摆位
- 显示中文拖拽安装提示

相关实现文件：

- `tools/package.sh`
- `tools/GenerateDMGBackground.swift`

补充说明：

- 本地正常打包时，会生成这套视觉化 DMG
- GitHub Actions 的发布 workflow 现在默认走 `--unsigned-dev --headless-dmg`
- WHY：CI 环境里不依赖 Finder 布局，产物更稳定，失败面更小
- 如果未来需要恢复视觉化 DMG，再评估是否切换到 self-hosted runner

## 建议的发布后检查

打包完成后，建议至少手工检查：

1. 安装包（DMG）能正常打开
2. `Caps Nav.app` 和 `Applications` 图标都存在
3. App 拖入“应用程序”后能正常启动
4. 首次启动后菜单栏图标正常出现
5. 权限申请、设置页和前缀键监听都能正常工作

如果你想额外确认架构，可以执行：

```bash
lipo -info '/path/to/Caps Nav.app/Contents/MacOS/Caps Nav'
```

预期输出中应同时出现：

- `arm64`
- `x86_64`

## 对外分发时的说明建议

如果你分发的是 `dev` 模式产物，需要提前告诉用户：

- 第一次打开时，macOS 可能拦截
- 需要到“系统设置 -> 隐私与安全性”中选择“仍要打开”
- 更准确地说，通常是在“系统设置 -> 隐私与安全性 -> 安全性”区域看到放行入口
- 如果用户看到“Apple 无法验证 `Caps Nav.app` 是否包含可能危害 Mac 安全或泄露隐私的恶意软件”，这属于当前预发布包的预期现象
- 它虽然是 `Release` 构建，但还不是经过 Apple 正式签名和公证的公开发布包

可以直接给用户这段操作说明：

1. 在拦截弹窗里先点“完成”
2. 打开“系统设置”
3. 进入“隐私与安全性”
4. 在“安全性”区域找到 `Caps Nav.app` 的提示
5. 点击“仍要打开”
6. 再次打开应用，并在确认弹窗中继续启动

如果你分发的是 `release` 模式产物，仍建议用户：

- 首次启动后先完成辅助功能权限授权
- 再去测试默认键位和配置方案功能
