# 发布与打包

这份文档描述 Caps Nav 当前的双模式打包方式：

- `dev`：本地打包
- `release`：正式发布用的签名、公证打包

## 目标产物

当前仓库生成的核心产物是：

- 一个 `Universal` 通用版 App
  - 同时包含 `arm64` 与 `x86_64`
- 一个可分发的 `.dmg`
- 一个对应的 `SHA-256` 摘要文件

这意味着：

- Apple Silicon Mac 可以直接使用
- Intel Mac 也可以使用
- 但前提仍然是目标系统满足当前工程的最低版本要求：`macOS 13.0+`

## 模式说明

### `dev` 模式

用途：

- 本地开发者打包
- 给测试同事或少量用户分发体验包

特点：

- 构建 `Release` 通用版 App
- 生成视觉化安装 DMG
- 不做 `Developer ID` 签名
- 不做 `Apple notarization`

### `release` 模式

用途：

- 正式发布
- 对外分发前的标准签名与公证链路

特点：

- 先走 `archive/exportArchive`
- 使用 `Developer ID Application` 导出签名 App
- 对最终 DMG 做 `notarytool submit`
- `stapler staple` 贴票据
- 最后做 Gatekeeper 验证

## 前提条件

执行打包脚本前，建议先确认：

- 已安装 Xcode Command Line Tools
- 本机可以正常执行 `xcodebuild`
- 当前工程能本地构建成功
- 使用的 macOS 环境自带 `hdiutil`、`ditto`、`shasum`
- 本机可正常执行 `swift` 与 `osascript`

如果你要执行 `release` 模式，还需要额外确认：

- 机器钥匙串里已安装可用的 `Developer ID Application` 证书
- 已配置 `xcrun notarytool` 的 keychain profile
- 本机可用 `codesign`、`xcrun`、`spctl`

## 一条命令打包

### 本地开发打包

在仓库根目录执行：

```bash
./tools/package-release.sh --mode dev
```

脚本会自动完成：

1. 构建 `Release`
2. 强制生成 `arm64 + x86_64` 通用包
3. 校验产物确实包含双架构
4. 生成带背景图和 Finder 布局的视觉化 `.dmg`
5. 生成 `.dmg.sha256`

### 正式发布打包

```bash
./tools/package-release.sh --mode release \
  --signing-identity "Developer ID Application: Example Corp (TEAM123456)" \
  --team-id "TEAM123456" \
  --notary-profile "capsnav-release"
```

脚本会自动完成：

1. `xcodebuild archive`
2. 生成 `ExportOptions.plist`
3. `xcodebuild -exportArchive` 导出 `Developer ID` 签名 App
4. 校验导出 App 的通用架构与签名
5. 生成带背景图和 Finder 布局的最终 `.dmg`
6. 调用 `xcrun notarytool submit --wait`
7. `xcrun stapler staple`
8. `xcrun stapler validate` 与 `spctl` 最终验证
9. 生成 `.dmg.sha256`

## 可选参数

### 自定义输出目录

```bash
./tools/package-release.sh --mode dev --output-dir /absolute/path/to/output
```

默认输出目录是：

```text
<repo>/artifacts/dev/
```

正式发布模式默认输出目录是：

```text
<repo>/artifacts/release/
```

### 自定义 DerivedData 目录

```bash
./tools/package-release.sh --mode dev --derived-data /absolute/path/to/.build/ReleasePackaging
```

默认构建目录是：

```text
<repo>/.build/ReleasePackaging
```

### 跳过 SHA-256 文件

```bash
./tools/package-release.sh --mode dev --skip-sha256
```

## `release` 模式参数来源

`release` 模式必须拿到这三个值：

- `signing identity`
- `team id`
- `notary profile`

脚本读取优先级是：

1. 命令行参数
2. 环境变量
3. 缺失时报错退出

对应环境变量：

```bash
export CAPS_NAV_SIGNING_IDENTITY="Developer ID Application: Example Corp (TEAM123456)"
export CAPS_NAV_TEAM_ID="TEAM123456"
export CAPS_NAV_NOTARY_PROFILE="capsnav-release"
```

然后可直接执行：

```bash
./tools/package-release.sh --mode release
```

## notary profile 建议配置

建议先在发布机器上保存一次凭据：

```bash
xcrun notarytool store-credentials "capsnav-release" \
  --apple-id "your-account@example.com" \
  --team-id "TEAM123456" \
  --password "app-specific-password"
```

之后脚本里只传 `--notary-profile capsnav-release`，不要把 Apple ID 和密码直接写进脚本或仓库文档。

## 产物命名

当前产物名格式为：

```text
Caps-Nav-<版本号>-<构建号>-macOS-universal.dmg
```

例如：

```text
Caps-Nav-1.0-1-macOS-universal.dmg
```

对应摘要文件：

```text
Caps-Nav-1.0-1-macOS-universal.dmg.sha256
```

## DMG 内容

生成的 DMG 内默认包含：

- `Caps Nav.app`
- `Applications` 快捷方式

这是一种常见的 macOS 安装分发形式，用户打开后可以直接把 App 拖到“应用程序”目录。

## 视觉化安装盘

当前 DMG 已经不再是默认文件夹视图，而是会额外做这几层处理：

- 自动生成一张品牌化背景图
- 设置 Finder 窗口尺寸
- 固定 `Caps Nav.app` 与 `Applications` 的摆位
- 用中文安装提示引导用户拖拽安装

相关实现文件：

- `tools/package-release.sh`
- `tools/GenerateDMGBackground.swift`

视觉方向约束见：

- `docs/design/界面与配色规范.md`

## 产物验证建议

打包完成后，建议至少手工检查这几项：

1. 双击打开 DMG 是否正常
2. DMG 内是否能看到 `Caps Nav.app` 和 `Applications`
3. 把 App 拖到“应用程序”后能否正常启动
4. 首次启动后状态栏图标是否正常出现
5. 权限申请、设置页打开、前缀键监听是否正常

如果要额外确认架构，也可以执行：

```bash
lipo -info '/path/to/Caps Nav.app/Contents/MacOS/Caps Nav'
```

预期输出里应同时出现：

- `arm64`
- `x86_64`

## 对外分发时要说明的限制

如果你使用的是 `dev` 模式，对外分发时要提前告诉用户：

- 第一次打开时，macOS 可能会拦截
- 用户需要到“系统设置 -> 隐私与安全性”里点击“仍要打开”
- 之后再授予辅助功能权限，Caps Nav 才能正常工作

补充一点：

- 如果你是在受限沙箱、远程执行器或某些 CI 环境里运行脚本，`hdiutil` 可能无法拉起磁盘映像服务
- 这时脚本会直接提示你改在本机终端执行
- 这属于系统环境限制，不是脚本本身的打包逻辑错误
