# 事件流转

## 从启动到监听

1. `CapsNavApp` 创建 `AppBootstrap`。
2. `AppBootstrap.start()` 创建 `~/Library/Application Support/Caps Nav/` 和 `profiles/`。
3. `SettingsStore` 保证 `settings.json` 存在。
4. `ProfileLoader` 保证 `default.json` 存在并加载全部 profile。
5. `ProfileManager` 选择 active profile。
6. `PermissionManager` 刷新辅助功能权限。
7. 已授权时，`KeyEventInterceptor.start()` 创建 `CGEventTap`。

## 从按键到动作

1. App 启动监听前，`PrefixKeyRouter` 优先把 `Caps Lock` 临时映射成 `F18`。
2. 用户按下物理 `Caps Lock`，系统对 App 暴露为 `F18 keyDown`。
3. `KeyEventInterceptor` 识别 prefix key down，并交给 `PrefixStateManager` 设为 active。
4. 用户继续按下字母键。
5. `KeyEventInterceptor` 解析出 `Trigger`，并在 active profile 中查找映射。
6. 找到映射后，`ActionResolver` 把 `output` 转成 `KeyStroke`。
7. `EventEmitter` 发送合成键盘事件。
8. 用户松开物理 `Caps Lock`，系统对 App 暴露为 `F18 keyUp`。
9. `PrefixStateManager` 设为 inactive，同时悬浮提示展示按下/松开状态。

## 当前风险点

- runtime remap 的生命周期需要管理好，正常退出时必须恢复原始映射。
- 当前 `EventEmitter` 采用“带 flags 的主键 down/up”实现，能满足第一阶段验证，但后续可能要升级为“修饰键显式按下/抬起”模型。
- 输入法候选状态、Terminal、IDE、网页编辑器需要单独回归测试。
