# ADR 0002: Runtime Caps Remap To F18

## 状态

Accepted

## 决策

运行时将物理 `Caps Lock` 临时重映射为 `F18`，并把 `F18` 作为内部 prefix 事件源处理。

## 背景

仅靠普通 `CGEventTap` 很难可靠阻止 `Caps Lock` 的系统锁定语义，而且 `maskAlphaShift` 也不适合表达“物理按下/松开”。

用户实测已经验证了两个问题：

- 系统大小写切换仍然发生
- 松开后 prefix 状态无法可靠恢复

## 方案比较

### 继续直接监听 Raw Caps

- 优点：实现简单
- 缺点：系统 Caps 行为难挡住，按下/松开语义不稳定

### 运行时 Remap Caps -> F18

- 优点：F18 是普通按键，`keyDown/keyUp` 语义稳定，原始 Caps 锁定行为不再触发
- 缺点：实现复杂度更高，需要管理 remap 生命周期

## 为什么选择它

这是当前最符合产品目标的工程实现路径。用户体验要的是“按住式 prefix”，不是“切换式锁定键”，因此先把物理键转换成普通键，再做事件拦截，是更稳的底层模型。

## 当前残余风险

- 如果 App 被强制杀掉，运行时 remap 可能在本次登录会话里暂时残留，通常重新启动 App 或重新登录即可恢复。
- 后续可以再补 crash recovery 和 remap state 备份，进一步收敛这个风险。
