# Caps Nav

Caps Nav 是一个 macOS 原生菜单栏工具。它把 `Caps Lock` 固定变成前缀键，让你在主键区完成光标移动、文本选择、删词、到行首/行尾等常见编辑动作，减少手离开字母区去找方向键的频率。

这是一个面向“高频文本编辑”场景的增强工具，不是窗口管理器，也不是通用自动化平台。它更像是一层稳定、可配置、长期可用的键盘导航层。

## 项目要解决什么问题

很多人在写代码、写文档、写 Markdown、处理表格或后台内容时，会频繁做这些动作：

- 左右上下移动光标
- 按词移动
- 选中一个字符、一整段词或到行首行尾
- 删除一个字符、一个单词

这些操作在 macOS 上虽然都有系统快捷键，但分散在方向键、`Option`、`Command`、`Shift` 等组合里，长期使用时会让手频繁离开主键区。

Caps Nav 的思路很简单：

- `Caps Lock` 固定是前缀键
- 按住 `Caps Lock`
- 再按字母键
- 软件把这次输入转换成对应的编辑动作

这样就能把常见导航动作稳定收拢到主键区附近。

## 这不是一个什么项目

为了让产品边界清晰，Caps Nav 当前明确 **不做** 下面这些方向：

- 不做窗口管理
- 不做通用脚本自动化平台
- 不做前缀键自定义
- 不做 App 专属配置切换
- 不做多步宏
- 不做云同步

## 当前已经实现的能力

截至当前仓库状态，Caps Nav 已经具备这些可用能力：

- 菜单栏常驻应用
- 启动后默认只保留状态栏图标，不主动弹出设置页或权限窗口
- 支持 `随系统 / 亮色 / 暗色` 三种主题模式
- 支持多种状态栏图标样式，并可在设置中切换
  - 包括默认、实心、快捷键、Caps、Caps 实心、Command、CN 字标、导航字标、ES 主键区
- 支持应用级运行总开关，暂停后恢复 `Caps Lock` 原生行为
- 支持可选全局开关快捷键，用于在任意前台应用中启用或暂停 Caps Nav
- 状态栏菜单可直接启用/暂停 Caps Nav，暂停态图标会显示额外标识
- 辅助功能权限检查与重新检测
- Caps 短按回退到系统默认大小写切换
- Caps 短按阈值可配置，默认 `200 ms`
- 按住 Caps 时显示悬浮帮助框
- 悬浮帮助框支持 `顶部 / 左侧 / 右侧`
- 悬浮帮助框默认显示在 `右侧`，并会根据屏幕尺寸自动调整宽高，尽量完整展示全部说明
- 悬浮帮助框支持整体透明度调节
- 悬浮帮助框能高亮当前实际按下的映射键
- 快捷键练习窗口，支持 `认键练习 / 连招挑战`
- 多配置方案管理
- 配置方案新增、复制、重命名、删除
- 配置方案拖拽排序
- 图形化按键映射编辑
- 默认方案映射只读，避免误改内置基线
- 内置动作支持“功能效果”悬停动态预览，并可自动播放或手动切换前后状态
- 开机自启开关
- 打开应用数据目录、配置目录、定位 settings 文件
- 自定义 About 窗口、权限说明窗口

## 工作方式

Caps Nav 的运行链路可以概括为：

1. App 常驻在菜单栏。
2. 启动时读取 `settings.json` 和配置方案文件。
3. 已授予辅助功能权限后，开始监听全局键盘事件。
4. 用户按住 `Caps Lock`。
5. 用户继续按一个映射键，例如 `s`。
6. 软件根据当前激活的配置方案，把 `Caps + s` 解析成某个内置动作，例如“光标向左移动”。
7. 软件向当前应用重发对应的系统编辑快捷键。

用户只需要把它理解成：

- `Caps Lock` = 前缀键
- 按住它，再按定义好的字母键

底层实现细节不需要参与日常使用。

## 快捷键练习

为了帮助用户更快熟悉当前配置方案里的键位，Caps Nav 现在提供了一个独立的“快捷键练习”窗口。

当前第一版包含两种模式：

- `认键练习`
  - 更适合第一次熟悉键位
  - 会按当前激活配置方案随机出题，重点帮助你建立“动作 -> 答案键”的对应关系
- `连招挑战`
  - 更适合已经知道大致键位后的提速训练
  - 反馈节奏更快，更强调正确率、连击和反应时间

训练特性：

- 题目直接来自当前激活的配置方案
- 训练窗口会展示当前方案的键位总览
- 会统计本轮正确率、平均反应时间、最佳连击和易错项
- 支持从菜单栏、设置页概览和 App 菜单打开

输入识别说明：

- 已授予辅助功能权限时，会尽量按真实使用手感来识别 `Caps Lock + 触发键`
- 未授予辅助功能权限时，训练窗口会退化为“先按一次 Caps，再输入答案键”的本地模拟方式

当前限制：

- 只做单轮训练统计，暂不持久化历史记录
- 目前聚焦键位熟悉和节奏训练，不做文本输入实战关卡

## 权限说明

Caps Nav 的核心能力依赖 macOS 的 **辅助功能权限**。

为什么需要：

- 监听全局键盘事件
- 稳定识别前缀键按下与松开
- 拦截原始按键，避免字母漏出到当前输入框
- 把目标动作重发为系统编辑快捷键

如果没有这个权限：

- App 界面仍然可以打开
- 运行状态会显示为“等待授权”
- 但全局键盘监听、按键拦截、悬浮提示和快捷键重发无法正常工作
- 状态栏入口、设置页、配置方案管理和应用级启用/暂停仍然可以使用

## Caps 短按回退机制

为了尽量兼顾前缀键和系统原生大小写切换，Caps Nav 支持“短按回退”：

- 默认阈值：`200 ms`
- 如果 `Caps Lock` 在阈值内按下立刻松开
- 并且期间没有触发其他按键映射
- 就回退执行系统原生的大小写切换

设置规则：

- 阈值必须是大于等于 `0` 的整数
- `0` 表示完全关闭短按回退，Caps 只作为前缀键使用

## 配置方案

Caps Nav 采用 **配置方案（Profile）** 驱动：

- 一个配置方案对应一份 JSON 文件
- 可以有多套方案
- 当前激活方案会持久化保存
- 切换方案后，悬浮提示和键位映射立即生效

当前支持：

- 新增配置方案
- 复制已有配置方案
- 重命名自定义配置方案
- 删除自定义配置方案
- 拖拽排序配置方案
- 右键定位单个配置方案对应的 JSON 文件
- 编辑当前方案中的映射

补充规则：

- `默认方案` 作为基础方案保留，不能重命名，也不能删除
- 当激活的是 `默认方案` 时，下方“按键映射”列表为只读状态，不能新增、修改或删除映射
- 如果需要改默认键位，推荐先复制默认方案，再在副本上继续调整
- “功能效果”列会显示一个预览小图标，鼠标悬停后才展开浮层预览
- 内置动作会在浮层中循环演示文本沙盘动画，也支持手动切换“动作前 / 动作后”
- 如果某一行配置的是 `shortcut` 自定义快捷键，则浮层只展示会发送的快捷键说明，不伪造统一文本效果

## 按键映射

每条映射由两部分组成：

- `trigger`
  - 前缀键后的触发键，例如 `s`
- `output`
  - 目标动作，当前主要是内置编辑动作

示例：

```json
{
  "trigger": {
    "key": "f",
    "modifiers": []
  },
  "output": {
    "type": "builtin",
    "action": "moveRight"
  },
  "description": "光标向右移动"
}
```

这段 JSON 表示的是“一条完整的按键映射规则”。下面把每一个 key 都拆开说明。

### 最外层对象

最外层这一整段对象，表示“当用户按下某个前缀后的按键时，Caps Nav 应该执行什么动作”。

它当前主要由 3 个字段组成：

- `trigger`
- `output`
- `description`

### `trigger`

`trigger` 用来描述“这条映射是被什么按键触发的”。

示例：

```json
"trigger": {
  "key": "f",
  "modifiers": []
}
```

#### `trigger.key`

`key` 表示前缀键 `Caps Lock` 按住之后，用户接下来按下的那个主按键。

在这个示例里：

```json
"key": "f"
```

意思就是：

- 用户先按住 `Caps Lock`
- 然后再按 `f`
- 这条映射就会被命中

补充说明：

- 这里保存的是触发键本身，不包含 `Caps Lock`
- `Caps Lock` 是固定前缀键，不写进配置
- 当前实现里，这里通常是一个非修饰键，比如 `f`、`s`、`e`、`;` 这类常用按键

#### `trigger.modifiers`

`modifiers` 表示在按这个触发键时，额外同时按下的修饰键。

示例里：

```json
"modifiers": []
```

表示：

- 不需要额外修饰键
- 只要 `Caps Lock + f` 就能触发

如果未来某条映射需要额外修饰键，它会长这样：

```json
"modifiers": ["shift"]
```

表示：

- 必须按住 `Caps Lock + Shift + 某个键`

当前允许的修饰键值只有：

- `shift`
- `control`
- `option`
- `command`

### `output`

`output` 用来描述“这条映射命中之后，Caps Nav 应该执行什么结果”。

示例：

```json
"output": {
  "type": "builtin",
  "action": "moveRight"
}
```

#### `output.type`

`type` 表示输出动作的类型。

示例里：

```json
"type": "builtin"
```

意思是：

- 这条映射触发后，执行的是软件内置动作
- 不是用户自定义快捷键

当前支持两种类型：

- `builtin`
  - 内置动作，例如移动光标、选中、删除、到行首/行尾
- `shortcut`
  - 自定义快捷键输出，例如转发成某个系统快捷键

#### `output.action`

`action` 只在 `output.type = "builtin"` 时出现。

示例里：

```json
"action": "moveRight"
```

表示：

- 这条映射命中后，执行的内置动作是“向右移动”

也就是说，示例这条规则整体含义是：

- `Caps Lock + f`
- 执行“光标向右移动”

`action` 这里保存的是 **内置动作的标识名**，不是最终发给系统的按键字符。

例如：

- `moveLeft`
- `moveRight`
- `moveUp`
- `moveDown`
- `selectLeft`
- `deleteWordBackward`

Caps Nav 会在运行时把这些语义动作转换成真正的系统编辑快捷键。

比如 `moveRight` 最终会被转换成：

- `Right Arrow`

如果 `output.type` 是 `shortcut`，那这里就不会再有 `action`，而会换成 `shortcut` 对象，例如：

```json
"output": {
  "type": "shortcut",
  "shortcut": {
    "key": "a",
    "modifiers": ["control"]
  }
}
```

#### `output.shortcut`

`shortcut` 只在 `output.type = "shortcut"` 时出现。

它表示：

- 这条映射命中后，不再执行 Caps Nav 的内置语义动作
- 而是直接向系统发送一组你自定义的目标快捷键

也就是说：

- `trigger` 负责定义 `Caps Lock` 后你按什么
- `output.shortcut` 负责定义命中后要转发成什么快捷键

#### `output.shortcut.key`

`key` 表示目标快捷键里的主键。

例如：

```json
"key": "a"
```

表示这条映射最终会发送以 `A` 为主键的快捷键。

当前图形化编辑器里，这个主键可以从列表里选择，当前主要支持：

- 字母
- 数字
- 常用符号
- 方向键
- `Return`
- `Space`
- `Tab`
- `Escape`
- `Delete`
- `Forward Delete`

#### `output.shortcut.modifiers`

`modifiers` 表示目标快捷键里还要同时按下哪些修饰键。

例如：

```json
"modifiers": ["control"]
```

表示最终发送的是 `Control + A`，而不是单独一个 `A`。

它是一个数组，原因是一个快捷键可以带多个修饰键，比如：

```json
"modifiers": ["command", "shift"]
```

这就表示目标快捷键会变成 `Command + Shift + 主键`。

当前支持的修饰键有：

- `shift`
- `control`
- `option`
- `command`

### `description`

`description` 是这条映射的用户可见说明文案。

示例里：

```json
"description": "光标向右移动"
```

它的作用不是参与触发判断，而是给用户看。

当前主要会用在这些地方：

- 设置页中的“快捷键说明”
- 按住 Caps 时的悬浮帮助框
- 配置方案预览信息

也就是说：

- `trigger` 决定“怎么触发”
- `output` 决定“触发后做什么”
- `description` 决定“在界面上怎么把这件事说明给用户看”

补充说明：

- `description` 可以理解为这条映射的中文注释
- 它优先服务于可读性和悬浮提示体验
- 如果缺失，软件会回退到内置动作的默认说明文案

### 把这条示例完整翻译成人话

下面这条配置：

```json
{
  "trigger": {
    "key": "f",
    "modifiers": []
  },
  "output": {
    "type": "builtin",
    "action": "moveRight"
  },
  "description": "光标向右移动"
}
```

等价于一句话：

- 当用户按住 `Caps Lock` 再按 `f`，且没有额外修饰键时，Caps Nav 会执行“光标向右移动”这个内置动作，并在界面中把这条映射显示为“光标向右移动”。

当前设置页里，映射也可以直接图形化编辑：

- 快捷键列：选择触发键
- 功能列：选择内置动作，或者打开自定义快捷键面板
- 自定义快捷键面板：默认支持直接按键录入，也可以切换到列表选择主键与修饰键
- 快捷键说明列：自动显示当前说明文案，切换功能后会同步更新

## 本地存储位置

Caps Nav 会在下面这个目录下保存数据：

```text
~/Library/Application Support/Caps Nav/
```

目录结构大致如下：

```text
Caps Nav/
├── settings.json
└── profiles/
    ├── default.json
    ├── profile-xxxx.json
    └── ...
```

其中：

- `settings.json` 保存全局设置
- `profiles/` 保存各个配置方案

App 当前已经提供：

- 打开应用数据目录
- 打开配置方案目录
- 在 Finder 中定位 `settings.json`

## 系统要求

- macOS 13.0+
- 已授予辅助功能权限

## 从源码运行

### 方式一：使用 Xcode

1. 打开 `Caps Nav.xcodeproj`
2. 选择 `Caps Nav` scheme
3. 运行到本机 macOS

### 方式二：命令行构建

在项目根目录执行：

```bash
xcodebuild -project 'Caps Nav.xcodeproj' -scheme 'CapsNav' -configuration Debug -destination 'platform=macOS' -derivedDataPath .build/DerivedData build
```

## 打包与分发

如果你要把 Caps Nav 发给其他人使用，当前仓库已经提供了一条双模式打包脚本。

在仓库根目录执行：

```bash
./tools/package-release.sh --mode dev
```

`dev` 模式会自动生成：

- `Release` 通用版 `Caps Nav.app`
- 带背景图与安装引导布局的 `.dmg`
- 对应的 `SHA-256` 摘要文件

当前默认是：

- 同时支持 `arm64` 和 `x86_64`
- 最低系统版本仍然受工程配置约束，目前是 `macOS 13.0+`

需要注意：

- `dev` 模式不包含 `Developer ID` 与 `notarization`
- 所以发给别人后，第一次打开时仍可能被 macOS 安全策略拦截
- 用户需要到“系统设置 -> 隐私与安全性”中选择“仍要打开”

如果你要走正式发布链路，可以执行：

```bash
./tools/package-release.sh --mode release \
  --signing-identity "Developer ID Application: Example Corp (TEAM123456)" \
  --team-id "TEAM123456" \
  --notary-profile "capsnav-release"
```

## 项目结构

```text
Caps Nav/
├── App/                # 启动组装与应用总入口逻辑
├── Features/           # 设置页、菜单栏、悬浮提示、权限窗口、About 窗口
├── Infrastructure/     # settings/profile 读写、默认配置准备
├── Models/             # Profile、Mapping、AppSettings 等模型
├── Resources/          # 内置默认配置
├── Support/            # 主题、日志、JSON 编解码等支持代码
└── System/             # 全局事件监听、前缀状态、动作解析、事件重发
```

## 已知限制与注意事项

- 当前只支持 macOS。
- 前缀键固定是 `Caps Lock`，不是可配置项。
- 不同应用、输入框、终端、输入法候选态对方向键和删除键的解释可能存在差异。
- 这类工具天然依赖系统权限和全局事件链路，极端场景下仍需要做更多兼容性测试。

建议重点回归的应用包括：

- TextEdit
- Safari / Chrome 输入框
- VS Code
- Xcode
- Terminal / iTerm2
- 中文输入法候选框激活场景

## 路线边界

当前仓库明确坚持以下方向：

- 保持 `Caps Lock` 为固定前缀键
- 保持“编辑导航层增强器”的产品定位
- 优先做稳定性、可视化配置和产品体验，而不是把它扩展成复杂脚本平台
